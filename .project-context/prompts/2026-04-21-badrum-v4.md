# DEL I — Badrum v4 (feature)

---

## 1. Varför vi gör detta

Badrumsvyn i kalkylatorn ligger kvar som **schablon-only** (typ `'schablon'`) med två mutex-alternativ: "Helrenovering badrum standard" 85 000 kr och "Helrenovering badrum Plus" 115 000 kr. Det är en överlevare från v1 som vi satte i väntan på bättre underlag.

Nu har vi det bättre underlaget. Dennis körde en skarp Wikells-kalkyl för HSB-3a den 21 april 2026, plus att han skapat två egna byggdelar i Wikells för kakel respektive plastmatta på vägg (8.502, 8.503). En tredje — våtrumsfärg på vägg (8.504) — genererades programmatiskt från samma mönster och verifierades genom import i Wikells. Det ger oss riktiga kilopriser per material, inte gissade klumpsummor.

**Problemet med schablon-only ur kundens perspektiv:**

Tänk dig att en kund har ett 3 m²-gästbadrum och ett 8 m²-huvudbadrum. Båda är "standard" eller båda är "Plus"? Schablonen säger 85 000 oavsett. Men det 3 m²-rummet har kanske hälften så mycket kakelyta, hälften så mycket golv. Kalkylen blåser upp det lilla badrummet och underskattar det stora. Kunden märker inte det, men *du* märker det när du ska skriva riktig offert och upptäcker att schablonen var fel.

Vi vill istället att badrum ska räknas **per m²** på samma sätt som sovrum och hall — då kan kunden se hur mycket varje kvadratmeter kostar, välja vägg- och golvmaterial som i andra rum, och få ett pris som skalar med faktisk area.

**Ytterligare vinst:** Q9 i OPEN_QUESTIONS.md förbereder att varje post får `info: { description, image }` för tooltip. När badrum är per-post får tooltip-featuren naturliga hängare (kakel, plastmatta, våtrumsfärg, klinker på våtrumsgolv) istället för en klumpschablon som inte har något meningsfullt att förklara.

## 2. Nuvarande beteende

### I datamodellen (`src/data.js` rad 251–297)

Badrum är definierat som:

```js
badrum: {
  displayName: 'Badrum',
  defaultArea: 5,
  type: 'schablon',
  icon: ICON_BADRUM,
  items: [
    { id: 'badrum_standard', label: 'Helrenovering badrum standard', price: 85000, unit: 'schablon', group: 'badrum', primary: true,
      followups: [
        { id: 'badrum_standard__rivning', label: 'inkl. rivning av befintligt badrum', price: 5500, unit: 'schablon', defaultChecked: true },
        { id: 'badrum_standard__fallspackling', label: 'inkl. fallspackling våtrumsgolv', price: 9100, unit: 'schablon', defaultChecked: true }
      ]
    },
    { id: 'badrum_plus', label: 'Helrenovering badrum Plus', price: 115000, unit: 'schablon', group: 'badrum', primary: true,
      followups: [
        { id: 'badrum_plus__rivning', label: 'inkl. rivning av befintligt badrum', price: 7500, unit: 'schablon', defaultChecked: true },
        { id: 'badrum_plus__fallspackling', label: 'inkl. fallspackling våtrumsgolv', price: 13700, unit: 'schablon', defaultChecked: true }

    }
  ],
  roomFollowups: [SKYDDSTACKNING_FU]

```

### I UI:t

När man lägger till ett badrum ser man två kort sida vid sida — "standard" och "Plus" — med radioknappar (mutex via `group: 'badrum'`). Klickar man standard får man +85 000 kr och två kryssade följeposter (+5 500 + 9 100 = 99 600 kr totalt). Klickar man Plus istället får man 115 000 + 7 500 + 13 700 = 136 900 kr. Inga val av material, ingen yta-skalning, inget sätt att säga "vi vill bara plastmatta på vägg istället för kakel".

### Priskalibrering jämfört med verkligheten

Vi gjorde ett sanity-check mot Wikells HSB-3a för ett 5 m² badrum:

- Rivning (15.001 klinker + 15.010 kakel + 17.005/8/9 inredning): ~6 000 kr
- Fallspackling 15.013: ~11 400 kr (5 × 2 283)
- Klinker på golv 15.015: ~12 800 kr (5 × 2 555)
- Kakel på vägg 8.502: ~52 300 kr (21 m² × 2 492)
- WC-duschinredning 17.034: ~30 800 kr
- Badrumsinredning 17.032: ~9 300 kr
- Våtrumsdörr 16.059 (Dennis val, se §3.4): ~4 700 kr
- UE El: 8 000 kr
- UE VS: 12 000 kr
- Skyddstäckning: 46 kr
- **Total maxad (klinker + kakel + allt på): ~147 000 kr**

Nuvarande schablon "Plus" 136 900 kr ligger alltså ~10 000 kr under "riktig maxad badrum". Schablon "standard" 99 600 ligger långt under även för ett plastmatta-badrum. Det innebär att vi riskerar att **underprissätta** badrum i nuvarande version.

## 3. Förväntat beteende

### 3.1 — Badrum blir `type: 'per_post'`

Precis som sovrum, hall, vardagsrum. Ingen schablon-kort-vy, ingen mutex standard/Plus. Istället får rummet en items-lista där kunden ser alla poster och kryssar det som gäller.

### 3.2 — Ytterligare två tvingande följeposter som alltid är på

Nya idén är att **rivning och fallspackling är följeposter på rumsnivå** som alltid är kryssade, precis som skyddstäckning redan är. De läggs till i `roomFollowups`:

- **Rivning badrum inkl inredning** — 1 200 kr/m² golv, default on, `inheritsParentArea: false` (skala mot rummets golvyta, inte mot en underliggande post)
- **Fallspackling våtrumsgolv** — 2 283 kr/m² golv, default on (obligatorisk i våtrum, kan inte kryssas av i praktiken — men vi lämnar checkbox för transparens)

Motivering: i Wikells är båda separata byggdelar (15.013 + rivningsrader) som inte är bundna till vilket golvmaterial man väljer. Att ha dem som rumsföljeposter matchar det mentala mönstret och gör dem synliga i kalkylen.

### 3.6 — Legacy-migrering av sparade kalkyler

Användare som har sparat en kalkyl med `badrum_standard: { checked: true }` i localStorage måste kunna öppna sin kalkyl utan att den kraschar. Vi gör så här:

1. Vid `syncFollowups(rum)` eller `loadStateFromStorage()`: om `rum.typ === 'badrum'` och `rum.valda.badrum_standard` eller `rum.valda.badrum_plus` finns, **lämna det datat intakt** men rendera det inte. Nya posterna visas som okryssade. Lägg till en discrete meddelande-rad överst i rummet: *"Denna kalkyl är sparad före 2026-04-22 då badrum blev per-post. Välj om materialen för att få rätt pris."*
2. Inga poster ska krascha. Pristotalen ska fortsätta fungera (nya posterna är okryssade → 0 kr från den biten; gamla posterna ignoreras → 0 kr från den biten). Användaren får välja om och får då det korrekta priset.
3. Testa specifikt: ladda en kalkyl med `{ badrum_standard: {checked:true}, badrum_standard__rivning: {checked:true}, badrum_standard__fallspackling: {checked:true} }` — den ska rendera, meddelande visas, totalen i det rummet ska vara 0 kr tills användaren kryssar nya poster.

Alternativet (automatisk migrering av gamla schabloner till nya per-post-värden) är mer komplext och riskerar att silently ändra priset för kunden. Bättre att synligt meddela att det är en gammal kalkyl.

## 7. Intelligent koppling mellan poster och rumsreglage (KRITISKT)

Hela poängen med att gå från schablon till per-post är att priset ska **skala intelligent** med rummets verkliga egenskaper. Om de nya posterna inte kopplas till rätt reglage faller hela v4-arbetet. Därför är det här avsnittet tvingande — Claude Code måste verifiera varje punkt innan implementation.

### 7.1 — Väggposter måste använda existerande `wallCalc`-mekanism

Existerande datamodell har redan en flagga för poster som ska skala mot väggyta istället för golvyta: `wallCalc: true`. Den används bl.a. av `MALNING_VAGG` och kakel-poster i Hall/Toalett. Motsvarande kalkyl-funktion finns redan:

```js
// src/data.js (redan implementerat)
function calcWallArea(yta, takhojd) {
  return calcPerimeter(yta) * takhojd;  // = 4 × √yta × takhöjd
}
```

**Observera att §3.5 ovan använder strängen `useWallArea: true` i konstant-definitionerna** — det är felskrivet i promptformuleringen. Den korrekta, befintliga flaggan heter **`wallCalc: true`**. Använd `wallCalc: true` för `BADRUM_VAGG_KAKEL`, `BADRUM_VAGG_PLASTMATTA` och `BADRUM_VAGG_FARG`. Inför **inte** en ny duplicerad flagga. Om du hittar inkonsekvenser i befintlig kod (t.ex. att någon post har `useWallArea` istället) — flagga det i din plan men ändra inte utan att fråga.

Verifikation: när användaren ändrar takhöjden från 2,4 till 2,7 m i ett badrum som har Kakel på vägg kryssat, ska priset för kakel-posten öka med exakt faktorn 2,7/2,4 = 1,125.

### 7.2 — Golvposter skalar mot `rum.yta`

`BADRUM_GOLV_KLINKER` och `BADRUM_GOLV_PLASTMATTA` har `hasArea: true` och **ingen** `wallCalc`-flagga. Det innebär att de automatiskt skalas mot `rum.yta` via existerande `calcItemTotal`-logik. Ingen extra kod krävs här — men verifiera att `calcItemTotal` verkligen hittar arean från rummet om posten inte har egen `area`-override (vanlig checkad post utan individuell area-editering → ärv från `rum.yta`).

### 7.3 — Rumsföljeposterna rivning och fallspackling skalar mot `rum.yta`

Det här är det **kritiska nya fallet** som inte finns i nuvarande kod. Existerande rumsföljeposter (`SKYDDSTACKNING_FU`) har `unit: 'kr/rum'` — de är alltså fasta per rum. De nya `BADRUM_RIVNING_FU` och `BADRUM_FALLSPACKLING_FU` har `unit: 'kr/m²'` och ska skalas mot rummets golvyta.

`calcFollowupTotal` i nuvarande kod hanterar redan `inheritsParentArea` (ärv area från parent-post) och `inheritsReducesFloor` (area från parent minus reducesFloor-poster). Men vad händer när **ingen** parent finns (det är en rumsföljepost, inte en post-följepost) **och** `unit: 'kr/m²'`? Det är en ny kombination.

**Kräv av Claude Code i planen:** beskriv explicit hur `calcFollowupTotal` ska utökas/justeras för att `unit: 'kr/m²'` + ingen parent = skala mot `rum.yta`. En rimlig implementation är att lägga till en gren: om followup saknar parent och `unit === 'kr/m²'`, använd `rum.yta` som area. Men det är Claude Codes uppgift att specificera exakt vilken funktion som ändras och på vilken rad.

Verifikation: ändra rumsytan från 5 till 8 m² → rivning ska gå från 5 × 1 200 = 6 000 kr till 8 × 1 200 = 9 600 kr. Fallspackling från 11 415 till 18 264 kr.

### 7.4 — Reglagen (yta, takhöjd) måste trigga re-render av badrum

Badrum använder idag `type: 'schablon'` som inte har yta/takhöjd-reglage i rumskortet. När badrum blir `type: 'per_post'` ska det få **samma** reglage som sovrum, hall och vardagsrum:

- Yta-input (m²) med stepper-knappar
- Takhöjd-input (m) med stepper-knappar

Ändring av något av dessa reglage måste trigga `rerenderRoomBodyDOM(rum)` + `updateSubtotalDOM(rum)` så att:

1. Alla golv-poster räknar om (yta-baserade)
2. Alla vägg-poster räknar om (wallCalc → calcWallArea(nya yta, nya takhöjd))
3. Rumsföljeposterna rivning + fallspackling räknar om (kr/m² × nya yta)
4. Rum-subtotal uppdateras

Det här faller ut gratis om Claude Code använder samma rendering-mönster som sovrum/hall. **Men verifiera att `type: 'per_post'` i sig räcker för att reglagen ska dyka upp** — om det finns någon switch `if (type === 'sovrum' || type === 'hall' …)` som glömmer badrum, måste den utökas.

### 7.5 — UE- och inrednings-poster är st-baserade och ska ha stepper-kontroll

`BADRUM_WC_DUSCH`, `BADRUM_INREDNING`, `BADRUM_DORR`, `BADRUM_UE_EL`, `BADRUM_UE_VS` har alla `hasCount: true`. De ska renderas med count-stepper (samma som innerdörr i sovrum). Inget speciellt krav här, men bekräfta i planen att de renderas med stepper och inte som "kryssa av/på"-only.

### 7.6 — Sammanfattning: checklist för Claude Code i planen

Bekräfta explicit i implementationsplanen att:

- [ ] Väggposter använder `wallCalc: true` (befintlig flagga), inte `useWallArea`
- [ ] Golvposter skalar mot `rum.yta` via existerande mekanism (`hasArea: true`)
- [ ] Rumsföljeposter med `unit: 'kr/m²'` + ingen parent → skala mot `rum.yta`; beskriv exakt var/hur `calcFollowupTotal` utökas
- [ ] Yta- och takhöjd-reglage renderas i badrumskortet när `type === 'per_post'`
- [ ] Ändring av yta/takhöjd triggar `rerenderRoomBodyDOM` + `updateSubtotalDOM` för badrum
- [ ] Alla 10 items + 3 rumsföljeposter räknar om vid arealändring (kör scenario B i §4.3 som sanity-test)

Om någon av punkterna inte kan uppfyllas utan större refaktorering — flagga det i planen och föreslå åtgärd innan du kodar.

# DEL II — Bugfix #65: Sovrum får fel default-antal innerdörrar

## 1. Varför vi gör detta

När en kund lägger till sitt första sovrum förifylls posten "Ny innerdörr" med `count = 2`. Andra sovrum och senare får `count = 1`. Antagandet bakom var förmodligen att ett första sovrum i en lägenhet ofta är genomgångsrum med dörr både in och ut, men det stämmer inte i praktiken — i en normal svensk lägenhetsritning har varje sovrum exakt **en** dörr ut mot hall/korridor. Att förifylla 2 innerdörrar är därför fel default för nästan alla användningsfall.

**Problem ur användarens perspektiv:** Dennis testade appen som novis och lade till Sovrum 1. Dörrpriset visade 10 140 kr istället för 5 070 kr utan att han hade valt något. Det är 5 070 kr för mycket som gömmer sig som "hjälpsam pre-fill" och som kunden mycket väl kan missa att justera ner. Konsekvens: överprissatt kalkyl mot slutkund, eller — om kunden upptäcker det i offertfasen — förtroendeskada.

Dennis har tagit upp det i task #65. Det är en enkel, isolerad fix.

## 2. Nuvarande beteende

I `src/app.js` rad 1354–1357 finns logiken som körs när en användare klickar "Lägg till rum" och väljer sovrum:

```js
if (type === 'sovrum') {
  const prevSovrum = state.rum.filter((r) => r.typ === 'sovrum').length;
  // Första sovrummet: 2 innerdörrar förifyllt. Övriga: 1 st.
  newRum.valda.innerdorr = { checked: true, count: prevSovrum === 0 ? 2 : 1 };
}
```

Kommentaren bekräftar att det var avsiktligt kodat så — inte en slump. Men logiken är fel.

Effekt i UI: Första sovrummet öppnar med "Ny innerdörr" redan ikryssad och count = 2. Pris visas som 10 140 kr (2 × 5 070).

## 3. Förväntat beteende

**Alla sovrum får `count = 1` förifyllt.** Kunden ser "Ny innerdörr" ikryssad med antal 1 och priset 5 070 kr. Om ett sovrum ska ha fler än 1 dörr (sällsynt) ändrar kunden själv.

Motivering för att fortfarande ha dörren **ikryssad som default**: De flesta sovrum renoveras med byte av innerdörr. Om kunden vill behålla befintlig dörr (måla den) kryssar hen av den nya dörren och kryssar istället "Måla befintlig innerdörr". Den friktionen är minimal och default-på är rätt för vanligaste fallet.

**Bonusuppstädning (rekommenderas men inte tvingande):** Flytta defaultvärdena från hårdkodning i `app.js` (rad 1354–1372) in i en deklarativ struktur i `src/data.js` — ett fält `defaultOnCreate: { itemId: {checked, count/area} }` per rumstyp. Det är exakt det mönster som Del I §3.5 kräver för badrum. Om du inför det mönstret för badrum bör du samtidigt migrera sovrum, entré och övrigt så att all default-logik bor på samma ställe. Annars blir det spretigt och svårt att underhålla.

Om du gör bonusuppstädningen: ta bort hela `if (type === 'sovrum') { … } else if (type === 'entre') { … } else if (type === 'ovrigt') { … }`-blocket i app.js och ersätt med en generisk loop som läser `ROOM_TYPES[type].defaultOnCreate` och applicerar värdena. Även `perRoom`-logiken för övrigt-rummet (som multiplicerar med antal befintliga rum) behöver bevaras — den kan vara kvar som specialfall eller tolkas deklarativt med ett flaggfält.

## 4. Verifiering

1. Öppna appen i incognito (tomt localStorage).
2. Klicka "Lägg till rum" → Sovrum. Första sovrummet.
3. Kontrollera: "Ny innerdörr" är ikryssad, antalsfältet visar **1**, priset visar **5 070 kr**.
4. Klicka "Lägg till rum" → Sovrum igen. Andra sovrummet.
5. Samma verifiering: antal = 1, pris = 5 070 kr.
6. Testa för säkerhets skull entré och övrigt-rummet också — dessa ska fortfarande funka som förut (klinker 2 m² förifyllt i entré; perRoom-skalning i Övrigt). Om du gjorde bonusuppstädningen, testa extra noggrant att inget förifylls fel.

# DEL III — Bugfix #66: Följepost-rader ska reagera på klick på hela raden, inte bara checkboxen

## 1. Varför vi gör detta

Följeposter (indenterade rader under en post, t.ex. "inkl. rivning av befintligt golv" under Ekparkett) renderas som en rad med tre element: indenterings-tecken, checkbox, etikett, pris. Klick-handlern är bara kopplad till checkboxens lilla `<span class="checkbox">`. Om användaren siktar på etiketten "inkl. rivning av befintligt golv" eller på prissiffran och missar checkbox-spannen — inget händer.

Dennis testade detta som novis och beskrev det som **friktion**: man försöker kryssa av rivningen, missar den lilla checkbox-rutan, och måste göra om. Det är inte en bugg som kraschar appen, men det är en småfriktion som upprepar sig många gånger i en komplett kalkyl (varje rum har flera följeposter). Samlad irritation över en 5-rums-kalkyl blir påtaglig.

**Mobil är värre än desktop.** På iPad/mobil är checkbox-rutan ~18 px bred — inom "fingertoppsmål-zonen" men klart mindre än vad Apple och Google rekommenderar (44 × 44 px). Att hela raden blir klickbar löser både träffytan och förstärker mentalmodellen "raden är ett val".

## 2. Nuvarande beteende

I `src/app.js` rad 264–278 (typ A-följepost, indenterad under parent) och rad 281–299 (typ B-följepost, rumsskopad):

```js
// Typ A
return `
  <div class="followup-row${checked ? '' : ' is-unchecked'}" data-followup-id="${...}">
    <span class="followup-indent" aria-hidden="true">└</span>
    <span class="checkbox${checked ? ' checked' : ''}" data-action="toggle-followup" data-followup-id="${...}"></span>
    <span class="followup-label">${escapeHtml(fu.label)}</span>
    <span class="item-price">${formatKr(displayPrice)}</span>
  </div>
`;
```

Bara `<span class="checkbox">` har `data-action="toggle-followup"`. Delegerad event-listener i rad 1277 fångar `e.target.closest('[data-action]')` vilket alltså bara matchar inom checkboxens 18 px.

## 3. Förväntat beteende

**Klick någonstans på följepost-raden ska toggla följeposten.** Undantag: om raden är dimmad/disabled (`is-disabled` eller `is-dimmed` för typ B som väntar på trigger) ska klick inte göra något.

**Implementationsstrategi — välj en:**

**Alternativ A (enkelt, rekommenderat):** Flytta `data-action="toggle-followup"` och `data-followup-id` från `<span class="checkbox">` upp till den omslutande `<div class="followup-row">`. Då fångar event-delegationen hela raden automatiskt utan att ändra lyssnarens kod. Ta bort `data-action` från checkbox-spannen (den ärver från parent via `closest`). Detta fungerar för båda typ A och typ B. För typ B: om `is-disabled`-klassen finns, hoppa över i handlern (lägg till check på `action.classList.contains('is-disabled')` eller checka att trigger-villkoret är uppfyllt i handlern själv — det är redan delvis logik i `toggleFollowup`).

**Alternativ B (mer kod, mer explicit):** Håll kvar checkbox-handlern men lägg till en ny lyssnare på `.followup-row` som kollar om klicket landade utanför `data-action`-elementet och manuellt triggar samma toggle-logik. Ger mer kontroll men dubblerad logik.

Välj **Alternativ A** — mindre kod, mindre risk för regression, samma mönster kan generaliseras till vanliga items om det visar sig behövas senare.

**Ska vanliga item-rader (icke-följeposter) också få samma behandling?** Dennis har inte rapporterat friktion där, så håll fokus. *Lämna item-raderna som de är.* Anledningen: item-raderna har input-fält för count/area och stepper-knappar — klick på hela raden kräver extra logik för att inte trigga toggle när användaren försöker klicka i input-fältet eller stepper. Följeposterna har inga sådana element (bara indent + checkbox + label + pris). Säkrare att bara fixa följeposter nu och lämna items för eventuell framtida iteration.

**CSS-bonus:** Lägg till `cursor: pointer;` på `.followup-row:not(.is-disabled):not(.is-dimmed)` så att hovervisuellt feedback visar att raden är klickbar. Ta bort `cursor: pointer` från `.followup-row .checkbox` om det finns där (annars blir det dubblat men skadar inte).

## 4. Verifiering

1. Ladda appen, lägg till ett sovrum.
2. Kryssa Ekparkett. Följeposterna "inkl. rivning av befintligt golv" och "inkl. flytspackling undergolv" (eller motsvarande enligt gällande default) blir synliga.
3. Klicka på **etiketten** (texten) "inkl. rivning av befintligt golv" — inte på checkbox-rutan. Följeposten ska togglas.
4. Klicka på **prissiffran** till höger på samma rad. Följeposten ska togglas igen.
5. Klicka på **indenterings-tecknet** (`└`) till vänster. Ska också toggla. Om inte — godkänt, eftersom `└` är tekniskt svårt att klicka på i praktiken.
6. Hovra över raden på desktop — cursor ska bli pekande hand (om CSS-bonusen gjordes).
7. Testa typ B (rumsskopad följepost, t.ex. Skyddstäckning i sovrum): kryssa på målning → skyddstäckning blir aktiv → klick på etiketten ska toggla.
8. Testa dimmad typ B: öppna ett rum utan att kryssa något målnings-item. Skyddstäckning är dimmad. Klick på raden ska **inte** toggla.
9. Mobilsimulering (DevTools iPhone 12): klicka på etiketten med "fingret". Ska toggla utan problem.
